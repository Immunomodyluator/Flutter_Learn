# 7. Управление состоянием

## 1. Суть концепции

**Состояние (State)** — данные, которые могут изменяться и влиять на UI. Управление состоянием — решение вопросов: где хранить данные, как их изменять и как уведомлять UI об изменениях.

---

## 2. Типы состояния

| Тип | Описание | Пример |
|-----|----------|--------|
| **Ephemeral (локальное)** | Нужно только одному виджету | Видимость пароля, выбранная вкладка |
| **App state (глобальное)** | Нужно нескольким виджетам | Авторизация, корзина, тема |
| **Server state** | Данные с сервера с кешированием | Список товаров, профиль |

---

## 3. Эволюция подходов

```
setState()
    ↓ (простое, но ограничено одним виджетом)
InheritedWidget
    ↓ (низкоуровневый, многословный)
Provider (uses InheritedWidget under the hood)
    ↓ (простой, хорош для среднего размера)
Riverpod
    ↓ (типобезопасный, без BuildContext)
Bloc/Cubit
    ↓ (предсказуемый, хорош для сложной логики)
GetX / MobX
    ↓ (реактивный, минимум кода, но спорный)
```

---

## 4. Выбор инструмента

```
Простое приложение (1-5 экранов)    → setState + поднятие состояния
Среднее приложение                   → Provider или Riverpod
Сложное приложение                   → Riverpod или Bloc
Нужна реактивность                   → Riverpod или MobX
Команда знает Redux                  → Bloc
```

---

## 5. Поднятие состояния (State Lifting)

Когда два виджета-сиблинга нуждаются в одних данных — поднимаем состояние к ближайшему общему предку.

```dart
// ❌ Каждый виджет хранит своё — рассинхронизировано
class CartIcon extends StatefulWidget { int count; }
class CartPage extends StatefulWidget { List items; }

// ✅ Родитель хранит состояние, передаёт дочерним
class ShopScreen extends StatefulWidget {
  @override
  State<ShopScreen> createState() => _ShopScreenState();
}

class _ShopScreenState extends State<ShopScreen> {
  final List<Product> _cart = [];

  void _addToCart(Product p) => setState(() => _cart.add(p));
  void _removeFromCart(Product p) => setState(() => _cart.remove(p));

  @override
  Widget build(BuildContext context) {
    return Row(
      children: [
        ProductList(onAdd: _addToCart),
        CartPanel(items: _cart, onRemove: _removeFromCart),
      ],
    );
  }
}
```

---

## 6. Когда поднятие не работает

Если состояние нужно в далёких частях дерева (например, корзина доступна в AppBar, в списке товаров и на странице оплаты одновременно), поднятие приводит к "prop drilling" — передаче данных через много уровней. В этом случае нужен менеджер состояния.

---

## 7. Общие принципы любого менеджера состояния

1. **Единственный источник правды** — данные хранятся в одном месте.
2. **Односторонний поток данных** — изменения идут от события → к состоянию → к UI.
3. **UI = f(State)** — UI полностью определяется состоянием.
4. **Разделение UI и бизнес-логики** — виджеты только отображают данные.

---

## 8. Практические рекомендации

1. **Начинай с `setState`** — не усложняй без необходимости.
2. **Поднимай состояние** когда двум сиблингам нужны одни данные.
3. **Добавляй Provider/Riverpod** когда "prop drilling" становится проблемой.
4. **Не переусложняй** — маленькое приложение не требует Bloc + Clean Architecture.
5. **Консистентность в команде** — важнее выбрать один подход и придерживаться его.
