# 1.4 ООП в Dart

## 1. Суть концепции

Относится к: **ООП, классы, наследование, интерфейсы, миксины**.

Dart — полностью объектно-ориентированный язык: всё есть объект, включая `int`, `bool`, функции. Классы поддерживают одиночное наследование, но Dart решает проблему множественного наследования через **миксины** (`mixin`). Любой класс неявно является интерфейсом — нет отдельного ключевого слова `interface`.

---

## 2. Как используется во Flutter

- `StatelessWidget` / `StatefulWidget` — базовые классы Flutter, от которых ты наследуешься
- `ChangeNotifier` — базовый класс для state management через `mixin` или наследование
- `SingleTickerProviderStateMixin` — добавляет функциональность через mixin (без наследования)
- Абстрактные классы — интерфейсы репозиториев в Clean Architecture
- `factory` конструktor — паттерн Singleton, фабричный метод

## 3. Синтаксис и базовый пример

### Классы и конструкторы

```dart
class User {
  final String id;
  final String name;
  int age;

  // Обычный конструктор (инициализирующий формальный параметр)
  User({required this.id, required this.name, required this.age});

  // Named constructor
  User.anonymous() : id = 'anon', name = 'Anonymous', age = 0;

  // Метод
  String get displayName => '$name ($age)';

  // Переопределение toString
  @override
  String toString() => 'User($id, $name)';
}

// Использование
final user = User(id: '1', name: 'Ivan', age: 28);
final anon = User.anonymous();
```

### Наследование

```dart
class Animal {
  final String name;
  Animal(this.name);

  String speak() => '...';
}

class Dog extends Animal {
  Dog(super.name); // передаём аргумент родительскому конструктору

  @override
  String speak() => 'Woof!';

class Cat extends Animal {
  Cat(super.name);

  @override
  String speak() => 'Meow!';

```

### Абстрактные классы (интерфейсы)

```dart
// Абстрактный класс — нельзя создать экземпляр, используется как контракт
abstract class Repository<T> {
  Future<List<T>> getAll();
  Future<T?> getById(String id);
  Future<void> save(T item);
  Future<void> delete(String id);
}

// Реализация
class UserRepository implements Repository<User> {
  @override
  Future<List<User>> getAll() async => []; // реальная логика

  @override
  Future<User?> getById(String id) async => null;

  @override
  Future<void> save(User item) async {}

  @override
  Future<void> delete(String id) async {}
}
```

### Миксины

```dart
// mixin — набор методов без состояния (обычно)
mixin Loggable {
  void log(String message) => print('[${runtimeType}] $message');
}

mixin Serializable {
  Map<String, dynamic> toJson();
}

// Класс может использовать несколько миксинов
class UserService with Loggable {
  Future<void> createUser(String name) async {
    log('Creating user: $name'); // метод из Loggable
    // ...
  }
}

// on — ограничение: mixin можно применять только к определённым классам
mixin AnimationMixin on State {
  late AnimationController controller;
  // ...
}
```

### factory конструктор

```dart
class Config {
  static Config? _instance;
  final String baseUrl;

  Config._internal(this.baseUrl);

  // Singleton через factory
  factory Config() => _instance ??= Config._internal('https://api.example.com');
}

// Всегда возвращает один объект
final c1 = Config();
final c2 = Config();
print(identical(c1, c2)); // true
```

### Интерфейсы через `implements`

```dart
// В Dart любой класс может выступить интерфейсом
class Printable {
  void print() {}
}

class Document implements Printable {
  @override
  void print() => debugPrint('Printing document...');
}
```

## 4. Реальный пример из Flutter

```dart
// Абстрактный репозиторий — контракт (интерфейс)
abstract class AuthRepository {
  Future<User> signIn(String email, String password);
  Future<void> signOut();
  Stream<User?> get authStateChanges;
}

// Конкретная реализация
class FirebaseAuthRepository implements AuthRepository {
  final _auth = FirebaseAuth.instance;

  @override
  Future<User> signIn(String email, String password) async {
    final credential = await _auth.signInWithEmailAndPassword(
      email: email,
      password: password,
    );
    return User(id: credential.user!.uid, name: email, age: 0);
  }

  @override
  Future<void> signOut() => _auth.signOut();

  @override
  Stream<User?> get authStateChanges => _auth.authStateChanges().map(
        (fbUser) => fbUser == null ? null : User(id: fbUser.uid, name: fbUser.email ?? '', age: 0),
      );
}

// ViewModel/Notifier с миксином из Flutter
class AuthNotifier extends ChangeNotifier {
  final AuthRepository _repository;
  User? _user;
  bool _isLoading = false;

  AuthNotifier(this._repository);

  User? get user => _user;
  bool get isLoading => _isLoading;

  Future<void> signIn(String email, String password) async {
    _isLoading = true;
    notifyListeners(); // уведомляем виджеты об изменении

    try {
      _user = await _repository.signIn(email, password);
    } finally {
      _isLoading = false;
      notifyListeners();
    }
  }
}
```

`AuthRepository` — абстракция. Виджеты и `AuthNotifier` зависят от интерфейса, а не от конкретного Firebase. При тестировании подменяем на `MockAuthRepository`.

## 5. Что происходит под капотом

### Implicit interfaces

В Dart **нет ключевого слова `interface`**. Любой класс автоматически является контрактом:

```dart
class Logger {
  void log(String msg) => print(msg);
}

// Реализуем Logger как интерфейс
class MockLogger implements Logger {
  final logs = <String>[];

  @override
  void log(String msg) => logs.add(msg);
}
```

### Миксины vs наследование

- `extends` — наследует реализацию (методы + поля). Только один.
- `implements` — реализует контракт (нужно переопределить всё). Можно несколько.
- `with` — добавляет готовую реализацию (mixin). Можно несколько. Нет конфликта diamond.

### vtable и dispatch

Dart использует vtable (таблица виртуальных методов) для полиморфизма. При вызове на базовом типе — всегда вызывается реализация конкретного объекта. Это стандартный OOP dispatch, без boxing.

## 6. Типичные ошибки

| Ошибка                                | Почему возникает                                            | Правильно                                                |
| ------------------------------------- | ----------------------------------------------------------- | -------------------------------------------------------- |
| Наследование вместо интерфейса        | Жёсткая связь, трудно тестировать                           | `implements` абстрактный класс для инверсии зависимостей |
| `mixin` с изменяемым состоянием       | Миксины с полями делают логику непредсказуемой              | Держи в mixin только поведение (методы)                  |
| Не переопределён `==` и `hashCode`    | Два `User('1', 'Ivan')` не равны даже с одинаковыми данными | Переопределяй или используй `freezed`                    |
| `factory` без `_private` конструктора | Можно создать несколько экземпляров                         | Закрывай основной конструктор: `MyClass._()`             |
| Глубокое дерево наследования          | Хрупкий код, сложно рефакторить                             | Предпочитай композицию наследованию                      |

## 7. Практические рекомендации

1. **Репозитории через `abstract class`** — стандарт Clean Architecture во Flutter. Тесты подменяют реализацию на мок.
2. **`ChangeNotifier` через `extends`, не `with`** — обычно наследуют, а не используют как mixin. `with ChangeNotifier` тоже возможно, но редко нужно.
3. **Миксины для кросс-срезающей функциональности** — логирование, сериализация, анимации: `with SingleTickerProviderStateMixin`.
4. **`freezed` для data-классов** — автогенерирует `==`, `hashCode`, `copyWith`, `fromJson/toJson`. Не пиши их вручную.
5. **`@immutable` аннотация** для классов-моделей — компилятор проверит, что все поля `final`.
6. **Не злоупотребляй наследованием** для виджетов — предпочитай композицию (виджет содержит другой виджет, а не наследует его).
7. **Named constructors** для разных способов создания объекта: `User.fromJson(map)`, `User.empty()`, `User.fromFirebase(doc)`.
