# 15. Производительность Flutter-приложений

## 1. Суть

Flutter нацелен на 60/120 fps, но неоптимальный код может вызвать «джанки» (jank) — пропущенные кадры, видимые как подтормаживания. Производительность определяется на трёх уровнях:

1. **Rendering** — сколько времени занимает построение и отрисовка кадра.
2. **Memory** — утечки памяти, избыточные объекты.
3. **Startup / Bundle** — холодный старт, размер APK/IPA.

---

## 2. Метрика: 16 мс на кадр

При 60 fps Flutter должен строить и рисовать каждый кадр за **≤16 мс** (при 120 fps — ≤8 мс).

```
Frame time = Build time + Layout time + Paint time + GPU composite
```

Если хотя бы один кадр превышает 16 мс → пользователь видит «залипание».

---

## 3. Главные источники проблем

| Проблема                     | Симптом                                          | Решение                                   |
| ---------------------------- | ------------------------------------------------ | ----------------------------------------- |
| Лишние `build()`             | Весь экран перестраивается при смене одного поля | `const`, `Selector`, `Consumer`           |
| Тяжёлые операции в `build()` | Жёсткая нагрузка на UI поток                     | Выносить в `compute()` / `Isolate`        |
| `setState` на корне дерева   | Перестройка всего приложения                     | Локальный `setState` или state management |
| Неправильные изображения     | Декодирование в UI потоке                        | `ResizeImage`, `cacheWidth`               |
| Не переиспользуются виджеты  | Создаются заново на каждом кадре                 | `const`-виджеты, `RepaintBoundary`        |

## 4. Инструменты

| Инструмент                      | Для чего                         |
| ------------------------------- | -------------------------------- |
| **Flutter DevTools**            | Profiling, Timeline, Memory, CPU |
| `debugPrintRebuildDirtyWidgets` | Логирование всех rebuild         |
| `debugRepaintRainbowEnabled`    | Визуализация перерисовок         |
| `flutter build --analyze-size`  | Анализ размера APK/IPA           |
| `dart:developer` Timeline       | Кастомные метрики                |

---

## 5. Режимы сборки

```bash
# Debug — много проверок, медленно (нельзя измерять производительность!)
flutter run

# Profile — близко к Release, с DevTools-инструментами
flutter run --profile

# Release — максимальная производительность, без инструментов
flutter build apk --release
```

**Важно**: всегда измеряй в `--profile` или `--release` режиме на реальном устройстве.

## 7. Рекомендации

1. **Профилируй сначала** — не оптимизируй вслепую. DevTools покажет реальное узкое место.
2. **`const` везде где возможно** — самый простой способ ускорить rebuild.
3. **`Profile mode`** на реальном устройстве — не на эмуляторе.
4. **Изображения правильного размера** — не декодировать 4K изображение в 50px аватар.
5. **Тяжёлые CPU операции** (парсинг JSON, шифрование) → `compute()` / `Isolate`.
