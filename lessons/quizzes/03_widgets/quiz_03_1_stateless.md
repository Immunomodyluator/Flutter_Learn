# Квиз: StatelessWidget

### Вопрос 1 🟢

Какой метод обязательно нужно переопределить в `StatelessWidget`?

- A) `build()`
- B) `render()`
- C) `create()`
- D) `paint()`

<details>
<summary>Ответ</summary>

**A) `build()`**

`build(BuildContext context)` — единственный абстрактный метод `StatelessWidget`. Он возвращает `Widget` (дерево виджетов). Вызывается каждый раз, когда Flutter решает, что виджет нужно перестроить.

</details>

### Вопрос 2 🟢

Когда происходит перестройка (rebuild) `StatelessWidget`?

- A) Никогда — это статический виджет
- B) При изменении переданных параметров (props) родительским виджетом
- C) По таймеру каждые 16мс
- D) Только при Hot Reload

<details>
<summary>Ответ</summary>

**B) При изменении переданных параметров (props) родительским виджетом**

`StatelessWidget` перестраивается когда родитель вызывает `build()` с новыми аргументами. Flutter сравнивает ключи и типы виджетов для минимизации перестроений. Само по себе `StatelessWidget` не инициирует rebuild.

</details>

### Вопрос 3 🟢

Почему поля `StatelessWidget` должны быть `final`?

- A) Это требование Dart
- B) Виджет иммутабельный: один раз создан — данные не должны меняться
- C) Для ускорения компиляции
- D) Только рекомендация, не обязательно

<details>
<summary>Ответ</summary>

**B) Виджет иммутабельный: один раз создан — данные не должны меняться**

Flutter описание UI через неизменяемые объекты — ключевой принцип. Виджеты — лёгкие конфигурационные объекты. Если данные меняются — создаётся новый объект виджета, а не меняется старый. Линтер предупреждает о non-final полях в виджетах.

</details>

### Вопрос 4 🟡

Зачем нужен `BuildContext` в методе `build()`?

- A) Только для передачи стилей от родителя
- B) Для доступа к дереву виджетов: получать темы, локализацию, navigator, inherited данные
- C) BuildContext — ссылка на State объект
- D) Для идентификации виджета в дереве

<details>
<summary>Ответ</summary>

**B) Для доступа к дереву виджетов: получать темы, локализацию, navigator, inherited данные**

`BuildContext` — «ручка» к позиции виджета в дереве. Через него: `Theme.of(context)`, `Navigator.of(context)`, `MediaQuery.of(context)`, `Provider.of<T>(context)`. Контекст рендеренного виджета не совпадает с контекстом child виджетов.

</details>

### Вопрос 5 🟡

Что произойдёт, если `build()` вернёт разные типы виджетов при перестройке?

- A) Flutter выбросит исключение
- B) Flutter заменит старый элемент дерева новым — это нормально
- C) Flutter проигнорирует изменения
- D) Это невозможно — тип должен быть фиксирован

<details>
<summary>Ответ</summary>

**B) Flutter заменит старый элемент дерева новым — это нормально**

Flutter diff-алгоритм сравнивает тип и ключ. Если тип виджета изменился — старый element удаляется, новый создаётся с нуля. Это работает корректно, но предпочтительнее скрывать/показывать через `Visibility` или `if` с одинаковым типом.

</details>

### Вопрос 6 🟡

Для чего нужен `const` конструктор в `StatelessWidget`?

- A) Обязательное требование
- B) Позволяет Flutter кэшировать экземпляр и не пересоздавать его при перестройке родителя
- C) Ускоряет компиляцию
- D) Запрещает передавать параметры

<details>
<summary>Ответ</summary>

**B) Позволяет Flutter кэшировать экземпляр и не пересоздавать его при перестройке родителя**

`const MyWidget()` — один и тот же объект в памяти во всём дереве (compile-time constant). Flutter видит `==` и пропускает перестройку. Это одна из самых эффективных оптимизаций. Требует, чтобы все поля были `final` и переданы как константы.

</details>

### Вопрос 7 🟡

Что такое «три дерева» Flutter?

- A) Widget, Element, RenderObject
- B) View, ViewModel, Model
- C) Layer, Node, Canvas
- D) BuildContext, State, Widget

<details>
<summary>Ответ</summary>

**A) Widget, Element, RenderObject**

- **Widget** — иммутабельное описание UI (конфигурация)
- **Element** — связующее: управляет жизненным циклом, хранит State, отслеживает дерево
- **RenderObject** — выполняет layout и paint

Widget дерево легковесно и часто пересоздаётся. Element и RenderObject — «дорогие», переиспользуются максимально.

</details>

### Вопрос 8 🔴

Что такое `@immutable` аннотация и как она связана с `StatelessWidget`?

- A) Она запрещает наследование
- B) Линтер-аннотация: все поля должны быть `final` (рекурсивно по иерархии)
- C) Обеспечивает thread-safety
- D) Используется только для тестирования

<details>
<summary>Ответ</summary>

**B) Линтер-аннотация: все поля должны быть `final` (рекурсивно по иерархии)**

`StatelessWidget` помечен `@immutable` из пакета `meta`. Линтер предупредит, если поля нодного из классов иерархии не финальны. Это не runtime-проверка, а статический анализ. Поддерживает принцип иммутабельности виджетов.

</details>

### Вопрос 9 🔴

Что происходит в Element дереве, когда `StatelessWidget` перестраивается?

- A) Element создаётся заново
- B) Element вызывает `widget.build(this)` с новым конфигурационным объектом — Element остаётся тем же
- C) Element и RenderObject создаются заново
- D) Ничего — Element кэширует прошлый build

<details>
<summary>Ответ</summary>

**B) Element вызывает `widget.build(this)` с новым конфигурационным объектом — Element остаётся тем же**

Flutter отделяет Widget (конфигурацию) от Element (живого объекта в дереве). При rebuild родитель создаёт новый Widget объект, но Flutter сравнивает тип виджета и **сохраняет** существующий Element, передавая ему новый Widget. Element вызывает `build()` заново.

</details>

### Вопрос 10 🔴

Почему вызов `context.findAncestorWidgetOfExactType<T>()` антипаттерн?

- A) Метод не существует
- B) Линейный поиск вверх по дереву O(n), не уведомляет при изменении — лучше использовать InheritedWidget
- C) Работает только в debug режиме
- D) Возвращает не тот тип

<details>
<summary>Ответ</summary>

**B) Линейный поиск вверх по дереву O(n), не уведомляет при изменении — лучше использовать InheritedWidget**

`findAncestorWidgetOfExactType` — O(глубина дерева). `InheritedWidget` регистрирует подписки O(1) и автоматически уведомляет зависимые виджеты. Для передачи данных по дереву нужны `InheritedWidget`, `Provider`, `Riverpod` — они оптимизированы для этого.

</details>
