# Квиз: Null Safety в Dart

**Тема:** 01.5 — Null Safety  
**Вопросов:** 10 | 🟢 1–3 · 🟡 4–7 · 🔴 8–10

---

### Вопрос 1 🟢

Как объявить переменную, которая может содержать `null`?

- A) `int x = null`
- B) `int? x`
- C) `nullable int x`
- D) `Optional<int> x`

<details>
<summary>Ответ</summary>

**B) `int? x`**

Знак `?` после типа делает его nullable. По умолчанию все типы в Dart — non-nullable (начиная с Dart 2.12). `int? x` может быть `null`, `int x` — никогда.

</details>

### Вопрос 2 🟢

Что делает оператор `??`?

- A) Проверяет равенство с null
- B) Возвращает левый операнд, если он не null, иначе — правый
- C) Бросает исключение если значение null
- D) Конвертирует null в пустую строку

<details>
<summary>Ответ</summary>

**B) Возвращает левый операнд, если он не null, иначе — правый**

`String name = user?.name ?? 'Гость'` — если `user?.name` равно null, вернётся `'Гость'`. Используется для задания значений по умолчанию для nullable выражений.

</details>

### Вопрос 3 🟢

Что произойдёт при использовании `!` (null assertion) на null значении?

- A) Возвращается `null`
- B) Ошибка компиляции
- C) Бросается `Null check operator used on a null value` в runtime
- D) Возвращается значение по умолчанию для типа

<details>
<summary>Ответ</summary>

**C) Бросается `Null check operator used on a null value` в runtime**

`!` говорит компилятору: «Я уверен, что это не null». Если в runtime значение окажется null — будет брошен `TypeError`. Использовать `!` нужно только когда вы уверены в non-null значении логически, но компилятор это не может доказать.

</details>

### Вопрос 4 🟡

Что такое flow analysis в контексте null safety?

- A) Анализ производительности приложения
- B) Статический анализ, который автоматически сужает nullable тип до non-nullable в безопасном контексте
- C) Отслеживание потоков данных в Stream
- D) Анализ памяти

<details>
<summary>Ответ</summary>

**B) Статический анализ, который автоматически сужает nullable тип до non-nullable в безопасном контексте**

```dart
String? name;
if (name != null) {
  print(name.length); // здесь name автоматически String (не String?)
}
```

После проверки `!= null` анализатор понимает, что в этом блоке `name` точно не null и не требует `!` или `?.`.

</details>

### Вопрос 5 🟡

Что делает `?.` (null-aware оператор)?

- A) Создаёт nullable переменную
- B) Вызывает метод/свойство, если объект не null; иначе возвращает null
- C) Сравнивает с null
- D) Присваивает только если значение null

<details>
<summary>Ответ</summary>

**B) Вызывает метод/свойство, если объект не null; иначе возвращает null**

`user?.address?.city` — безопасная цепочка вызовов. Если любой элемент цепочки null — выражение возвращает null без NPE. Тип результата становится nullable.

</details>

### Вопрос 6 🟡

Что делает оператор `??=`?

- A) Сравнивает два nullable значения
- B) Присваивает значение переменной, только если она равна null
- C) Бросает исключение если значение уже не null
- D) Объединяет nullable типы

<details>
<summary>Ответ</summary>

**B) Присваивает значение переменной, только если она равна null**

`cache ??= expensiveComputation()` — вычисляет и присваивает `expensiveComputation()` только если `cache == null`. Это lazy initialization паттерн. Если `cache` уже не null — правая часть вообще не вычисляется.

</details>

### Вопрос 7 🟡

Почему использование `!` повсюду — плохая практика?

- A) Это медленнее по производительности
- B) Компилятор запрещает более 3 использований `!`
- C) Это переносит проверку типов с compile-time на runtime, увеличивая риск крашей
- D) `!` не работает с `final` переменными

<details>
<summary>Ответ</summary>

**C) Это переносит проверку типов с compile-time на runtime, увеличивая риск крашей**

Null safety даёт гарантии на этапе компиляции. Злоупотребление `!` отказывается от этих гарантий, фактически воспроизводя поведение старого кода без null safety. Лучше использовать `if (x != null)`, `x ?? default` или реструктурировать код.

</details>

### Вопрос 8 🔴

Что выведет код?

```dart
void main() {
  String? a;
  String b = a ?? (a = 'hello');
  print(a);
  print(b);
}
```

- A) `null` и `null`
- B) `hello` и `hello`
- C) `null` и `hello`
- D) `hello` и `null`

<details>
<summary>Ответ</summary>

**B) `hello` и `hello`**

`a ?? (a = 'hello')` — `a` равно `null`, поэтому вычисляется правая часть `(a = 'hello')`, которая присваивает `'hello'` переменной `a` и возвращает `'hello'`. Итог: `a = 'hello'`, `b = 'hello'`.

</details>

### Вопрос 9 🔴

Что такое `late final` и когда его использовать?

- A) Поле, которое инициализируется лениво и может быть присвоено только один раз
- B) Финальное поле, которое нельзя инициализировать в конструкторе
- C) Поле, видимое только в поздних версиях Dart
- D) Поле с отложенной компиляцией

<details>
<summary>Ответ</summary>

**A) Поле, которое инициализируется лениво и может быть присвоено только один раз**

`late final` объединяет ленивую инициализацию (`late`) с неизменяемостью после первого присвоения (`final`). Попытка присвоить значение во второй раз → `LateInitializationError`. Идеально для дорогостоящих вычислений, которые нужны не всегда.

</details>

### Вопрос 10 🔴

Как корректно обработать `List<String?>` и получить `List<String>` без null?

- A) `list.cast<String>()`
- B) `list.whereType<String>().toList()`
- C) `list.map((e) => e!).toList()`
- D) `list.removeWhere((e) => e == null)`

<details>
<summary>Ответ</summary>

**B) `list.whereType<String>().toList()`**

`whereType<String>()` фильтрует только элементы типа `String` (исключая `null`) и возвращает `Iterable<String>`. Вариант C (`e!`) бросит исключение если есть null. `cast<String>()` не фильтрует — бросает при обращении к null элементу.

</details>
